# Birdhouse Sensor Network — 새집형 산불 조기감지 노드 배치 시뮬레이션

산불 위험이 높은 곳이 아니라, 발화 후보지의 연기 이동 경로를 가장 잘 관측할 수
있는 지점에 새집형 카메라/센서 노드를 배치하기 위한 MVP 시뮬레이션이다.
`birdhouse_wildfire_sensor_placement_spec_minimal_data.md` 9.1절 범위를 구현했다.

## 분석 대상

- 의성군 (DEM 타일 7개: 36806/36807/36808/36810/36811/36812/36815)
- 봉화군 (DEM 타일 1개: 36803 — 다운로드된 범위만 분석됨)

## 실행

```bash
py -3 run_pipeline.py --region all      # web/data/uiseong.json, bonghwa.json 생성
py -3 serve_web.py                      # http://127.0.0.1:8010
```

`.env`에 `DATA_GO_KR_API_KEY`, `VWORLD_API_KEY`를 설정한다(`.env.example` 참고).

## 데이터 소스와 한계

| 소스 | 상태 | 비고 |
|---|---|---|
| 산림청 산불발생통계 | 실시간 | 위경도가 없어 지역·계절 위험 가중치로만 사용 |
| 기상청 ASOS 시간자료 | 실시간 | 봄철(3~5월) 12~18시 최근 3개 시즌 windrose |
| 에어코리아 대기질 | 실시간 | 측정소 좌표 미제공 API라 수작업 좌표 힌트로 최근접 매칭 |
| VWorld WFS | **현재 키로 INCORRECT_KEY** | 도로/건물/보호대상 서브스코어는 근사치로 대체(`approx: true`). VWorld 콘솔에서 WFS(2D 데이터) 서비스 활용신청 승인 여부를 확인하면, 재실행만으로 라이브 데이터로 전환됨 |
| DEM | 로컬 파일 | ascii xyz, EPSG:5186으로 추정해 역산 검증 완료 |

## 새집형 센서 실측 사양 반영

센서 실측 탐지 거리는 500m~1km다(초기 가정치였던 장거리 시야 카메라 모델은
폐기). 발화 지점 자체가 아니라 "시간이 지나며 번진 연기가 언제 센서 반경
안에 들어오는지"를 기준으로 탐지 시각을 계산한다(연기가 퍼질수록 유효 반경도
분당 25m씩 넓어진다고 가정). `pipeline/simulate.py`(즉석 시뮬레이션)와
`pipeline/camera_placement.py`(산별 그래프 최적화) 양쪽에 동일하게 적용된다.

## 산별 그래프 기반 최소 카메라 배치 (k-center)

카메라 배치는 "봉우리 두드러짐 + 발화후보지 근접도" 같은 휴리스틱 점수가
아니라, 그래프 알고리즘으로 "이 산에 카메라를 몇 개 놓아야 가장 늦게
탐지되는 지점까지도 목표 시간 안에 들어오는지"를 정량적으로 계산한다(휴리스틱
랭킹 방식은 시도해봤으나 센서 사거리가 500m~1km로 짧다는 사실이 확정되면서
"몇 개를 어디에"에 대한 확실한 답을 주지 못해 폐기했다).

```text
지역을 300m 격자 노드로 나눈다 (pipeline/fire_graph.py)
→ 두드러진 봉우리를 시드로 삼아 산 단위로 분할한다 (pipeline/mountains.py,
  비최대치 억제 방식 — 단순 프로미넌스+군집화는 능선을 타고 산 전체가
  하나로 뭉치는 문제가 있어 폐기)
→ 산마다 노드 전량 기준으로 경사 가중 Dijkstra를 돌려, 발화 시 다른 모든
  노드까지 도달 시간(분)을 구한다 (오르막은 빠르게, 내리막은 느리게)
→ greedy k-center(2-근사)로 "최악의 탐지 시간이 최소가 되는" 카메라를
  하나씩 추가한다 — 카메라는 실거리 500m~1km 이내를 즉시 커버 (pipeline/camera_placement.py)
→ 목표 20분 또는 산당 카메라 6개 상한에 도달하면 멈춘다
```

결과는 산마다 "카메라 1개→N개일 때 최악 탐지 시간" 순서 있는 목록이며,
`mountainCoverage` 키로 region JSON에 들어간다. 의성은 산 1,243개(그래프
추천 카메라 총 4,200개), 봉화는 190개(624개)로 분할됐다(각 산은 서로 독립된
서브그래프로 취급 — 불이 산 경계를 넘어 번지는 것은 다루지 않는다).

지도에는 두 단계로 보인다: 기본으로는 산마다 1순위 카메라 하나씩(옅은 점,
전체 분포 확인용)이 항상 떠 있고, 왼쪽 "산별 최소 카메라 배치"에서 특정
산을 고르면(또는 지도의 옅은 점을 클릭하면) 그 산의 경계와 전체 추천
카메라 순서가 강조 표시된다.

## 발화 지점 시뮬레이션 (임의 지점, 실시간 계산)

산림청 발화 "기록"이 있는 곳뿐 아니라, 아직 불이 난 적 없는 후보지에도 대비해야
하므로 오른쪽 "발화 지점 시뮬레이션" 카드에서 계절을 고르고 지도의 아무 지점이나
클릭하면:

- 그 지점에서 계절별 주풍향 분포에 따라 연기 경로가 어느 방향으로 퍼지는지
- 그래프 최적화로 나온 전체 추천 카메라(의성 4,200개/봉화 624개) 기준으로
  몇 분 만에 최초 탐지가 예상되는지
- 풍향별 확률 중 몇 %가 현재 카메라망으로 탐지되지 않는 사각지대인지

를 그 자리에서 계산해 지도 위에 표시한다. 이 계산은 정적 JSON이 아니라
`serve_web.py`의 `/api/simulate` 엔드포인트가 실시간으로 수행한다(DEM/추천
카메라 목록은 지역별로 메모리 캐시). 탐지 지연 3분은 카메라 스캔 주기+연기
판별 처리 시간에 대한 가정치이며, 실제 장비 사양이 정해지면 조정이 필요하다.

## 아키텍처

- `pipeline/` — 전부 표준 라이브러리만 사용(추가 pip 설치 불필요). DEM 파싱,
  TM 좌표변환, Shapefile 파싱, 그래프/Dijkstra까지 순수 파이썬으로 구현했다.
- `run_pipeline.py` — 지역별 배치 분석을 실행해 `web/data/*.json`을 만든다
  (발화 후보지 통계 + 4계절 windRose + 산별 그래프 최적화).
- `serve_web.py` — 정적 파일 서빙 + `/api/simulate` 실시간 시뮬레이션 API.
- `web/` — Leaflet 기반 정적 대시보드.

## 알려진 단순화

- 연기 이동은 유체 시뮬레이션이 아니라 기획서 6.3의 단순 규칙(풍속 구간별 고정
  길이 직선 경로)이다.
- 산불 확산 그래프는 경사만 반영하고 풍향은 반영하지 않는다(카메라 배치는
  바람이 어느 쪽으로 불든 최악의 경우를 방어하는 게 목적이라 등방향이 안전
  마진 측면에서 타당하다고 보고 단순화했다). 기준 확산속도(4m/min)는 봄철
  중간 조건 가정치이며 조정 가능하다.
- 카메라 후보 노드는 접근성(임도 거리 등) 제약 없이 순수 그래프 최적화
  결과다 — 실제 설치 가능성은 후속 필터링이 필요하다.
- VWorld WFS는 여전히 현재 키로 INCORRECT_KEY라 도로/건물 레이어는 쓰지
  않는다(파이프라인이 시도는 하되 실패해도 나머지 분석에는 영향 없음).
